# Node ↔ cloud protocol

This document describes the wire protocol between a HamVoipConfigGui
node (the Go `internal/cloudagent` package) and this cloud service (the
`/agent` WebSocket endpoint in `server/src/ws/agentServer.ts`), and how
this service bridges that connection to the browser's REST calls. It's
the authoritative reference for the message shapes both sides actually
send — the Go source (`internal/cloudagent/protocol.go` and its
`actions_*.go` files) and this repo's `ws/agentServer.ts` /
`services/relay.ts` are the ground truth if this drifts; open a PR
against whichever changed.

## Transport

- One WebSocket connection per device, at `wss://<cloud-host>/agent` in
  production (`ws://` only for local development — see SECURITY.md).
- The **node always dials out**; the cloud never opens an inbound
  connection to a node. This is the load-bearing design constraint:
  most nodes sit behind home NAT with no port forwarding, so there is
  no other direction this can work in.
- Exactly one connection is tracked per device at a time
  (`ws/agentServer.ts`'s `connections` map, keyed by device ID). A new
  connection from the same device implicitly replaces whatever was
  there before it (Node's own `ws` library closes the prior socket on
  key overwrite via garbage collection of the old reference plus its
  own eventual `close`/`error` firing — there's no explicit "only one
  connection per device" enforcement beyond the map having one slot).

## Connection lifecycle

1. Node dials the WS connection and immediately sends **hello**.
2. Cloud validates the API key and (if valid and the device is
   enabled) replies **helloAck** with `ok: true`; otherwise `ok: false`
   with an `error` string, then closes the connection.
3. Node starts a **heartbeat** loop (`system.status`, every ~20s) and a
   **read loop** that dispatches incoming `call`/`watch`/`unwatch`
   messages.
4. Either side can close at any time. The node's own reconnect loop
   (`internal/cloudagent/run.go`) retries with exponential backoff and
   full jitter (1s → 60s cap), resetting the backoff only after a
   *successful hello handshake* — a rejected API key backs off like any
   other repeated failure, not a fast fixed retry.
5. On reconnect, the cloud re-sends `watch` for whichever nodes still
   have an open live view in some browser (`resendWatchesForDevice`) —
   the node's own watch state is per-connection and resets to nothing
   on every fresh connect (`liveWatches.stopAll`), so this is what
   makes a live tab survive a reconnect without the browser having to
   notice and re-subscribe itself.

## Envelope

Every message, both directions, is a single JSON object over a text
frame:

```ts
interface Envelope {
  type: string; // "hello" | "helloAck" | "call" | "result" | "event" | "watch" | "unwatch"

  // hello (node -> cloud)
  apiKey?: string;
  nodes?: string[]; // node numbers this device currently reports (see config.Store.ListNodes)

  // helloAck (cloud -> node)
  ok?: boolean;
  error?: string;

  // call (cloud -> node) / result (node -> cloud)
  id?: string; // correlates a result back to its call
  action?: string;
  params?: unknown;
  data?: unknown;

  // event (node -> cloud)
  event?: string; // "status" | "nodeLive"

  // watch / unwatch (cloud -> node)
  node?: string; // node number
}
```

All fields are optional at the type level; which ones are populated
depends on `type` (and, for `event`, on `event`). This is one flat
struct rather than a discriminated union of several message types —
deliberately, so the same shape decodes trivially on both the Go side
(`encoding/json` into one struct) and the TypeScript side, at the cost
of most fields being irrelevant to any given message.

### hello (node → cloud)

Sent once, immediately after the connection opens.

```json
{ "type": "hello", "apiKey": "hvc_live_...", "nodes": ["2000", "2001"] }
```

`nodes` is a snapshot of `config.Store.ListNodes()` at connect time —
this becomes the device's cached `nodes` array in MongoDB (refreshed
again on every `status` heartbeat), never authoritative; the device
itself always is.

### helloAck (cloud → node)

```json
{ "type": "helloAck", "ok": true }
{ "type": "helloAck", "ok": false, "error": "invalid API key" }
{ "type": "helloAck", "ok": false, "error": "this device has been revoked" }
```

A rejected hello is followed by the cloud closing the connection.

### call (cloud → node) / result (node → cloud)

The RPC pair. `id` is a UUID generated by `services/relay.ts` per call
and must be echoed back unchanged in the matching `result`.

```json
{ "type": "call", "id": "b3f...", "action": "system.status", "params": null }
```

```json
{ "type": "result", "id": "b3f...", "ok": true, "data": { "...": "..." } }
{ "type": "result", "id": "b3f...", "ok": false, "error": "asterisk restart is disabled — turn it on in this node's own Cloud Sync settings first" }
```

`ok` is only ever explicitly `true` on success; a failed result omits
it (`omitempty` on the Go side), so treat "`ok` is not `true`" as
failure, not "`ok` is explicitly `false`". Every dispatched call —
known action or not, success or failure — is independently recorded on
both sides (see SECURITY.md's audit logging section), but the log
entries never include `params`, since several actions carry secrets
(a SkywarnPlus Pushover token, SA818 module settings).

### event (node → cloud)

Two kinds, both node-initiated, never correlated to a `call`:

```json
{ "type": "event", "event": "status", "data": { "asterisk_running": true, "uptime": "...", "hostname": "...", "error": "" } }
```

Pushed every ~20s while connected (plus once immediately on connect).
`data` is `internal/system.Snapshot`'s own JSON shape — see that Go
type for the authoritative field list. This is the **cheap, always-on**
heartbeat tier; it must stay small since it's real traffic over
someone's home uplink for every connected device, all the time.

```json
{ "type": "event", "event": "nodeLive", "node": "2000", "data": { "receiving": true, "signalOnInput": "...", "connected": [ /* rptstatus.ConnectedNode[] */ ] } }
```

Pushed every ~2s **only** for nodes the cloud has sent a `watch` for —
the **expensive, on-demand** tier, mirroring the local app's own
`internal/server/live.go` `liveHub` design. `connected` is
`rptstatus.ConnectedNode[]` — see that Go type for field names
(number, callsign, keyed, etc.).

### watch / unwatch (cloud → node)

```json
{ "type": "watch", "node": "2000" }
{ "type": "unwatch", "node": "2000" }
```

Sent by the cloud only while at least one browser has that node's live
view open (`nodeLiveHub.ts`'s subscriber count crossing 0→1 or 1→0),
never per-browser — the cloud side already deduplicates across however
many tabs are watching the same node. Both are idempotent on the node
side: watching an already-watched node, or unwatching one that isn't,
is a no-op.

## Action catalog

Every `action` name below is a fixed entry in `internal/cloudagent`'s
`actions()` map (`dispatch.go`) — this is an explicit allowlist, never
a generic "call this method by name" dispatcher (see SECURITY.md,
mitigation #3). An action name not in this table can never reach any
device-side call.

"Gate" is an additional check beyond a valid hello session — either a
local capability flag on the node (`AllowRemoteReboot` /
`AllowRawConfigEdit`, off by default, set on the node's own Cloud Sync
settings card) or, on this service's REST side, a step-up token (see
SECURITY.md). "none" means gated only by the device being connected at
all.

| Action | Params | Result | Gate |
|---|---|---|---|
| `system.status` | — | `system.Snapshot` | none |
| `system.restartAsterisk` | — | `{ok:true}` | `AllowRemoteReboot` + step-up |
| `system.reboot` | — | `{ok:true}` | `AllowRemoteReboot` + step-up |
| `config.listNodes` | — | `string[]` | none |
| `config.loadNode` | `{number}` | `config.Node` | none |
| `config.saveNode` | `config.Node` (create-or-update, idempotent) | `config.Node` | none |
| `config.deleteNode` | `{number}` | `{ok:true}` | none |
| `soundSchedule.list` | `{node}` | `soundschedule.Entry[]` | none |
| `soundSchedule.save` | `soundschedule.Entry` | `{ok:true}` | none |
| `soundSchedule.delete` | `{id}` | `{ok:true}` | none |
| `wxTone.list` | `{node}` | `wxtone.Entry[]` | none |
| `wxTone.save` | `wxtone.Entry` | `{ok:true}` | none |
| `wxTone.delete` | `{id}` | `{ok:true}` | none |
| `sa818.program` | `sa818.Settings` | `{ok, output}` | none (write-only hardware action; no capability flag — see actions_sa818.go) |
| `skywarn.listCounties` | — | `skywarnplus.CountyOption[]` (works even if SkywarnPlus isn't installed) | none |
| `skywarn.getStatus` | — | `skywarnplus.Status` | SkywarnPlus installed |
| `skywarn.setToggle` | `{key, value}` | `{ok:true}` | SkywarnPlus installed |
| `skywarn.setCounties` | `{codes: string[]}` (full replace) | `{ok:true}` | SkywarnPlus installed |
| `skywarn.addNode` | `{node}` | `{ok:true}` | SkywarnPlus installed |
| `skywarn.setPushover` | `{enable, userKey, apiToken, debug}` | `{ok:true}` | SkywarnPlus installed |
| `skywarn.setSkyDescribe` | `{apiKey, language, speed, voice, maxWords}` | `{ok:true}` | SkywarnPlus installed |
| `sounds.listAll` | — | `sounds.File[]` | none |
| `sounds.upload` | `{name, dataBase64}` | `{ok:true}` | none |
| `sounds.delete` | `{name}` | `{ok:true}` | none |
| `rawconfig.listFiles` | — | `string[]` (file names only, not sensitive) | none |
| `rawconfig.getFile` | `{file}` | `{sections: [{name, keys: [{key, value}]}]}` | `AllowRawConfigEdit` |
| `rawconfig.setKey` | `{file, section, index, value}` | `{ok:true}` | `AllowRawConfigEdit` + step-up |
| `rawconfig.addKey` | `{file, section, key, value}` | `{ok:true}` | `AllowRawConfigEdit` + step-up |
| `rawconfig.addSection` | `{file, section}` | `{ok:true}` | `AllowRawConfigEdit` + step-up |

`rawconfig.*`'s `file` must be one of `config.AllowedRawConfigFiles`
(`rpt.conf`, `iax.conf`, `usbradio.conf`, `simpleusb.conf`,
`voter.conf`, `extensions.conf`) — anything else is rejected before it
ever reaches a file read/write, on both the node and (redundantly) the
cloud's own route validation.

## REST ↔ WS bridging

The browser never speaks this protocol directly. Each domain's REST
route (`server/src/routes/*.routes.ts`) calls `sendAction(deviceId,
action, params, timeoutMs?)` (`services/relay.ts`), which:

1. Looks up the device's live connection (`ws/agentServer.ts`'s
   `getConnection`) — `503` (`RelayError`) if there isn't one.
2. Generates a UUID, sends a `call` envelope, and registers a pending
   promise keyed by that ID.
3. Resolves (or rejects) the promise when `handleMessage` in
   `agentServer.ts` sees a matching `result` — dispatched via
   `resolveCall`.
4. Times out after `timeoutMs` (default 10s; sound uploads get 30s) —
   `504`.
5. A device-returned `ok:false` result becomes a `400` (`RelayError`)
   — in this action set, a rejected call is consistently the caller's
   own bad input (a disabled capability flag, a disallowed raw config
   file, a bad node number), not an unexpected fault.

Routes that also need an audit trail (see SECURITY.md) call
`auditedSendAction(req, action, params, timeoutMs?)`
(`middleware/auditLogger.ts`) instead of `sendAction` directly — same
behavior, plus one `AuditLog` document per attempt.
